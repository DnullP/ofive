# 项目规划、专项和里程碑

## 迭代专项

### 后端专项

后端专项注重当前的后端接口质量

1. 面向前端的api接口保证集成测试覆盖，具有足够的仿真测试度
2. 接口性能具有基准测试，保证大规模笔记仓库场景下耗时可接受
3. 一致性强要求：当前后端模块基于真实文件系统中的笔记仓库建立了sqlite索引后，需要保证：
    - 文件和sqlite索引一致
    - 当文件发生变更时，索引能在短时间能同时更新
    - 当创建新文件、删除文件、移动文件时，需要同时更新索引

4. 对于解析类接口，需要提供各种场景的单元测试case来保证解析正确性
5. 梳理当前的后端接口、功能，评估前端的哪些逻辑、状态需要下沉到后端实现
6. 梳理后端对前端的事件通知机制，设置事件通知规范

#### 后端一致性治理计划

目标：逐步建立“后端持久态是全局读基准，前端编辑态是局部交互态”的一致性模型。

1. 定义状态分层规范
    - 明确 `persisted state` 表示真实文件系统中的已持久化状态，是所有读型组件的默认依据
    - 明确 `draft state` 表示前端编辑器中的未持久化状态，仅服务编辑器本身及被明确允许读取草稿态的局部交互
    - 禁止前端读型组件在未声明语义的前提下直接混用 persisted 与 draft

2. 建立后端读模型优先原则
    - Outline、知识图谱、搜索、Backlinks、文件元数据等读型组件逐步迁移为以后端接口返回结果为准
    - 前端对 Markdown 的结构化解析逐步下沉到后端，减少前端自定义规则与后端真实语义不一致的问题
    - 所有结构化读取接口补齐文档、日志和测试，作为前端统一语义来源

3. 建立“保存确认”优先于 watcher 的一致性链路
    - 前端主动保存成功时，以保存接口成功返回作为“持久态已更新”的主确认信号
    - 文件系统 watcher 事件主要用于处理外部修改、外部删除、外部重命名，以及作为兜底一致性校验机制
    - 降低前端主动写入后仍需等待 watcher 才能刷新读模型的时延

4. 建立后端事件通知规范
    - 区分“持久态变更通知”和“外部文件系统变更通知”两类事件，避免职责混杂
    - 统一事件负载字段，至少包含事件 id、来源 trace id、事件类型、目标路径、旧路径、时间信息
    - 规范事件命名、触发时机、去重策略和前端消费约束，减少事件风暴和竞态问题

5. 建立解析接口演进路线
    - 对“按路径解析文件”和“按内容解析草稿”两类接口做边界划分
    - 默认优先提供基于真实文件的读取接口，用于读一致场景
    - 仅在明确存在编辑态实时预览需求时，补充按内容解析接口，并明确其不是全局读基准

6. 建立一致性专项测试体系
    - 为保存、重命名、删除、移动、外部修改、索引更新等链路补齐集成测试
    - 针对大规模笔记库场景验证 watcher、索引、前端读取结果的一致性和延迟
    - 对 Markdown AST、outline、wikilink、graph 等解析类能力补齐覆盖 frontmatter、code fence、latex block、嵌套结构等测试用例

#### 后端事件与读模型里程碑

##### M1. 事件语义收敛

1. 梳理现有 Tauri 事件和前端内部事件的职责边界
2. 明确哪些事件属于跨边界事件，哪些只是前端内部协调事件
3. 输出后端事件规范文档，约束新增事件的命名、负载和消费方式
4. 收缩前端事件总线，仅保留确有必要的桥接职责

##### M2. Markdown 结构化读模型落地

1. 将 Markdown AST 接口定义为结构化读取基座能力
2. 评估是否补充更轻量的 Outline 专用接口，避免前端重复遍历完整 AST
3. 为 AST 或 Outline 结果补齐位置信息，支持标题导航、定位和增量刷新
4. 建立解析性能基准，确保在大笔记文件下可接受

##### M3. Outline 首个读一致改造样板

1. Outline 从前端本地 Markdown 正则解析迁移为调用后端 Markdown AST 或 Outline 接口
2. Outline 以当前聚焦文件路径和后端持久态结果为展示基准
3. Outline 在以下信号后刷新：
    - 当前聚焦文件切换
    - 当前文件保存成功
    - 当前文件收到外部文件系统变更事件
4. 当当前文件存在未保存修改时，UI 需明确标识“大纲基于已保存内容”或提供显式刷新说明
5. 为 Outline 增加前后端集成测试，覆盖 block 内伪标题、外部修改、保存后刷新等场景

##### M4. 读型组件统一迁移

1. 将知识图谱、搜索、Backlinks 等读型组件逐步迁移为以后端读模型为准
2. 清理前端重复解析 Markdown 的逻辑，减少规则分叉
3. 将前端 store 从“保存业务状态”逐步收敛为“缓存后端结果 + 维护局部交互态”
4. 明确各组件是否允许读取 draft，并形成统一约束

##### M5. 写入时延优化

1. 优化自动保存与手动保存的链路，尽量缩短用户编辑到文件落盘的时间
2. 为保存链路建立详细日志和耗时统计，定位延迟来源
3. 评估增量保存、批量合并、落盘节流等策略对一致性和体验的影响
4. 在保持持久态为基准的前提下，通过交互反馈弥补 unavoidable 的写入延迟

### 编辑器专项

#### 编辑器一致性专项

目标：编辑器作为唯一允许显式持有 draft 的核心模块，需要与后端持久态建立清晰的同步边界。

1. 明确编辑器状态边界
    - 编辑器内的实时文本、选区、组合输入、撤销栈等属于前端交互态
    - 编辑器对外暴露的“当前内容”不自动视为全局读基准
    - 编辑器与其他读型组件之间通过“保存确认后刷新 persisted 读模型”建立协同

2. 优化编辑器到持久态的同步路径
    - 自动保存策略以稳定、可观测、低延迟为目标
    - 保存成功后优先触发持久态刷新，而不是完全依赖 watcher 回流
    - 针对高频输入、连续保存、切换标签、失焦等场景补齐测试和日志

3. 明确编辑器周边 UI 的状态语义
    - Outline 等默认读取 persisted
    - 预览类或候选增强型组件若读取 draft，必须显式标注为“未保存预览”
    - 避免出现不同面板对同一笔记展示不同结构却无提示的情况


### UI体验专项

#### 一致性可感知体验改进

目标：当系统坚持以后端持久态为读基准时，必须让用户能感知“当前所见是否已保存”。

1. 提供明确的未保存状态提示
    - Tab、标题栏、Outline 等关键位置可感知当前文件是否存在未落盘修改
    - 避免用户误以为所有面板都在展示最新编辑中的内容

2. 提供持久态刷新反馈
    - 保存中、保存成功、保存失败、外部文件变更等状态需要有一致反馈
    - 对依赖 persisted 的面板，必要时提示“已根据最新文件刷新”

3. 处理延迟与冲突场景
    - 用户连续编辑、切换文件、外部工具修改同一文件时，需要有清晰反馈
    - 在一致性优先前提下，尽量减少“面板内容突然跳变”带来的困惑


### 插件化专项

#### 插件化阶段的状态与事件约束

目标：为未来插件系统建立统一的一致性和事件边界，避免插件绕过状态规范。

1. 插件默认读取后端持久态提供的标准接口，不直接依赖前端私有 store
2. 插件若需要草稿态能力，需要通过受控接口访问，并明确权限与语义
3. 插件事件体系需遵循核心事件规范，避免出现重复定义和语义冲突
4. 插件能力开放前，先沉淀核心读模型、保存确认、事件规范和日志体系

#### Outline 完全插件化改造计划

目标：将当前以“内建组件 + 注册式挂载”存在的 Outline，改造为真正的自注册插件模块，作为内容型面板插件的首个标准样板。

当前现状：

1. Outline 面板已经通过注册中心接入布局系统，具备插件式装配形态
2. Outline 组件本体仍位于内建布局模块中，与内建组件注册流程耦合
3. Outline 的数据获取与状态触发逻辑仍直接依赖前端当前实现，不具备插件级边界
4. 当前仓库已经存在 `src/plugins/` 自动发现和自注册示例，可作为迁移基础

改造目标：

1. 将 Outline 从 `layout` 内建模块迁出，成为 `src/plugins/` 下的独立插件模块
2. 通过插件模块自注册 activity/panel/i18n 资源，不再依赖 `registerBuiltinComponents`
3. 将 Outline 的读取逻辑统一到后端 persisted 读模型，成为“内容型读插件”标准样板
4. 把 Outline 对前端私有 store、布局内部细节、内建组件导出的耦合降到最低

##### Outline 插件化改造步骤

1. 插件边界收敛
    - 新建独立的 `outlinePlugin` 模块，负责自注册 Outline 的 activity 与 panel
    - 从内建注册入口中移除 Outline 的 activity/panel 注册逻辑
    - 将 Outline 所需的 i18n 资源从全局 locale 或内建组件依赖中收敛到插件自身

2. 插件 API 依赖收敛
    - 插件默认仅依赖公开 registry API、后端读取 API、必要的聚焦文件只读上下文
    - 不允许插件直接依赖布局私有实现或硬编码操作 Dockview 内部细节
    - 对插件需要的聚焦文件、打开文件、导航标题等能力，逐步抽象为稳定的插件侧接口

3. 数据模型改造
    - Outline 插件从前端本地 Markdown 正则解析迁移为调用后端 Markdown AST 或 Outline 接口
    - 以 persisted 结果作为默认展示数据源
    - 在未保存内容存在时，通过插件 UI 明确标注当前展示语义，而不是隐式混用 draft

4. 触发模型改造
    - 插件监听当前聚焦文件变化
    - 插件监听持久态内容更新信号，用于保存成功后的内容刷新
    - 插件监听外部文件系统变化，用于处理当前文件被外部修改、删除、移动的场景
    - 插件内部对刷新做去重、防抖和路径命中校验，避免重复读取

5. 导航能力补齐
    - 若 Outline 需要点击标题跳转，后端接口需返回标题位置信息
    - 插件侧通过稳定导航接口跳转，不直接依赖编辑器内部实现细节
    - 为后续其他内容型插件复用标题定位、文件定位能力打基础

6. 测试与回归保障
    - 为 Outline 插件增加独立单元测试和集成测试
    - 覆盖 frontmatter、code fence、latex block、外部修改、保存成功刷新、重命名/删除等场景
    - 验证插件化后不破坏现有布局、活动栏和右侧栏排序行为

#### 插件化专项需要补齐的核心能力

要让 Outline 真正成为插件，不只是把文件搬进 `src/plugins/`。插件化专项至少还需要补齐以下能力：

##### P1. 插件生命周期与注册规范

1. 定义插件模块的推荐目录结构、命名规范、自注册入口约定
2. 约束插件可以注册哪些对象：activity、panel、tab、setting、command、editor extension
3. 明确插件卸载、重复注册、冲突覆盖、排序优先级的规则
4. 为插件注册过程补充日志和错误隔离，避免单插件异常影响整个应用启动

##### P2. 插件公开 API 边界

1. 明确哪些能力属于插件可依赖的稳定 API，哪些是应用内部实现细节
2. 为插件提供读取 persisted 内容、读取聚焦文件、打开文件、导航、显示通知等公开接口
3. 禁止插件直接读写前端私有 store 的内部结构，减少后续重构成本
4. 对可能需要草稿态的能力建立受控 API，而不是让插件自行读取编辑器内存态

##### P3. 插件事件与状态模型

1. 为插件定义统一事件层次：底层文件事件、语义化 persisted 更新事件、局部 UI 事件
2. 明确插件默认应订阅哪一层事件，避免每个插件直接消费底层 watcher 细节
3. 建立插件与核心状态模型的契约，要求读型插件默认以后端 persisted 为准
4. 对跨插件通信需求进行收敛，避免重新引入松散的前端全局事件风格

##### P4. 插件设置与能力声明

1. 建立插件设置注册机制，让插件可以独立声明设置项、默认值和展示分组
2. 建立插件能力声明机制，例如是否需要文件读取、导航、命令注册、编辑器能力扩展
3. 为未来插件启停、按需加载、实验功能开关打基础

##### P5. 插件测试与质量门槛

1. 为插件模块建立最小质量标准：文档、日志、单元测试、必要的集成测试
2. 为插件注册、排序、启停、异常隔离建立测试基线
3. 对内容型插件要求验证其在大规模笔记仓库场景下的读取耗时与刷新正确性

##### P6. 插件迁移路线

1. 第一步：将 Outline 改造成首个完全插件化的内容型面板
2. 第二步：评估 Backlinks 插件，对其数据源和事件模型按 persisted 规则收敛
3. 第三步：将 Search、Graph 等读型面板逐步迁移到同一插件协议下
4. 第四步：梳理哪些现有“内建模块”适合继续保留内建，哪些应迁移为可插拔模块

#### 插件化专项里程碑

##### PM1. 插件系统基线完善

1. 完善插件注册规范、错误隔离、日志和测试基线
2. 输出插件稳定 API 清单和禁止依赖清单
3. 明确内容型插件的 persisted 读取约束

##### PM2. Outline 首个完全插件化样板

1. 将 Outline 迁移为 `src/plugins/` 下的自注册插件
2. 从内建注册入口中移除 Outline 的内建声明
3. 改造为基于后端 AST 或 Outline 接口的 persisted 读取模型
4. 补齐相关单元测试和集成测试

##### PM3. 内容型插件协议收敛

1. 抽象可复用的内容型插件数据刷新模型
2. 抽象统一的“聚焦文件变化 + persisted 更新 + 外部文件变化”触发协议
3. 为后续 Graph、Backlinks、Search 提供统一基础设施

##### PM4. 现有读型面板批量迁移

1. 迁移 Backlinks 到统一 persisted 读取与刷新协议
2. 迁移 Search/Graph 等读型面板到插件协议
3. 评估并清理不再需要的内建注册逻辑和重复解析实现

#### 注册系统分层开放计划

目标：梳理当前所有注册系统的动态开放边界，不追求“全部配置化”，而是建立“代码插件可全量扩展、声明式 contribution 可受控开放、宿主内部能力不外泄”的分层模型。

当前已存在的主要注册系统：

1. `activityRegistry`：注册 activity icon
2. `panelRegistry`：注册 sidebar panel
3. `tabComponentRegistry`：注册 tab 组件
4. `overlayRegistry`：注册全局 overlay / modal
5. `fileOpenerRegistry`：注册文件 opener
6. `convertibleViewRegistry`：注册 tab/panel 可转化视图
7. `sidebarHeaderActionRegistry`：注册侧栏 header action
8. `settingsRegistry`：注册设置页 section
9. `commandSystem`：注册命令与命令执行入口

开放分层原则：

1. 代码插件层：允许通过稳定 API 动态注册上述对象，作为最高能力层
2. 声明式 contribution 层：仅开放可序列化、可校验、可持久化的子集
3. 宿主内部执行层：不直接对配置或外部插件暴露 `render`、`resolveTab`、复杂回调和内部状态机

按扩展类型划分的目标结论：

1. 适合继续向声明式 contribution 开放：`activity`、受限 `panel`、`settings section`、`sidebar header action`、命令别名与绑定
2. 仅适合对代码插件开放：`tab component`、`file opener`、`convertible view`、自定义 `overlay render`
3. 不应继续向外开放：布局内部状态、拖拽规则、Dockview/Paneview 运行时细节、宿主私有 store 结构

##### PX1. Registry 盘点与分级

1. 为每个 registry 输出统一清单：注册对象、输入字段、是否含函数、是否依赖 React 组件、是否可序列化
2. 为每个 registry 标记开放级别：`declarative`、`code-only`、`host-internal`
3. 明确哪些 registry 已经具备稳定生命周期能力，哪些仍依赖宿主内部细节
4. 为每个 registry 补充示例，说明推荐使用场景和禁止用法

##### PX2. 统一 Contribution 模型

1. 设计统一的 contribution schema，先覆盖 `activity + panel-shell + command-binding`
2. 让静态插件注册和运行时配置新增共用同一套 contribution 到 registry 的转换逻辑
3. 统一 contribution 的校验、默认值填充、冲突处理和日志格式
4. 为 contribution 增加版本字段，给未来配置迁移留出空间

##### PX3. Activity / Panel 声明式开放

1. 将 activity 能力正式拆成“声明式 descriptor”与“执行式注册”两层
2. 定义 panel-shell 模型：允许创建默认容器型 panel，但不允许配置任意 render 函数
3. 让自定义 activity、未来用户插件清单、内置静态 activity 都复用同一转换链路
4. 为 activity/panel 的排序、默认位置、可见性和右键行为定义统一约束

##### PX4. Settings / Header Action 声明式开放

1. 为 settings section 设计“表单型 section contribution”，避免必须直接注册 React render 函数
2. 为 sidebar header action 设计“命令按钮型 contribution”，以命令绑定替代任意 onClick 回调
3. 明确哪些 settings/header action 仍保留为代码插件专用能力
4. 为这两类 contribution 补充宿主渲染规范和一致的 UI 约束

##### PX5. Code-only Registry 稳定化

1. 将 `tabComponentRegistry`、`fileOpenerRegistry`、`convertibleViewRegistry` 明确标记为 code-only 扩展点
2. 为这些 registry 输出最小稳定 ABI，减少对布局内部实现细节的耦合
3. 约束插件只能依赖公开 context，不允许直接依赖 Dockview 私有行为
4. 为 code-only registry 增加冲突检测和更清晰的错误日志

##### PX6. 宿主内部能力收口

1. 盘点当前暴露给插件的 context 字段，区分稳定 API 与临时兼容字段
2. 将真正属于宿主内部的布局状态、拖拽状态和 pane 运行时状态从“扩展面”里剥离
3. 为插件提供更高层、更语义化的宿主能力，例如打开文件、导航、执行命令、请求 modal
4. 建立“禁止直接依赖”的内部模块清单，写入插件开发规范

##### PX7. 文档与迁移路线

1. 在插件开发文档中补充 registry 分层说明和 contribution 设计原则
2. 为现有内置插件补一份迁移表：哪些已经符合分层模型，哪些需要重构
3. 先选 `activity/panel` 作为第一批 contribution 化样板，再评估 `settings/header action`
4. 在 contribution 模型稳定前，不继续扩大配置化范围到 `tab component`、`file opener`、`convertible view`

##### PX8. 测试与质量门槛

1. 为 contribution 解析、默认值、冲突覆盖、卸载清理建立单元测试
2. 为 registry 动态注册/注销后的 UI 行为建立集成测试
3. 为配置驱动 contribution 建立前后端持久化回归测试
4. 为 code-only registry 建立 ABI 稳定性测试，防止宿主重构破坏已有插件
