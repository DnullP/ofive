# ofive 后端并行开发架构目标与后续工作

## 1. 文档目的

本文档用于替代旧版“后端目标架构设计”作为当前后端规划依据。

当前后端规划同时关注几类目标：

- 合理分层
- 明确模块职责
- 区分公共模块与私有模块
- 支持多个 Agent 并行开发时尽量降低冲突
- 让公共依赖保持稳定、可治理、可版本化

其中，“多个 Agent 并行开发时不轻易发生代码冲突”只是评估标准之一，不是唯一标准。
新的规划要求后端最终能够同时满足：

- 多个 Agent 并行开发不同扩展功能
- 不轻易发生对同一公共模块的同时修改
- 新增功能尽量在各自边界内闭环完成
- 公共平台层保持稳定、低频变更、可版本化

换句话说，后端架构是否足够好，不能只看“模块是否拆开”，而要同时看：

- 分层是否合理
- 哪些模块应该作为稳定公共依赖
- 哪些模块应该作为扩展私有实现而避免被大量依赖
- 三个扩展功能能否同时开发
- 每个功能是否主要修改自己的模块边界
- 公共文件是否只在建立平台能力时少量变更，而不是每次扩展都要反复编辑

## 2. 核心评估标准

以后端是否同时满足“合理分层 + 清晰依赖边界 + 可并行开发”为准，采用以下标准判断架构是否达标。

### 2.1 分层合理性

后端分层仍然是第一原则之一，因为并行开发能力本身依赖于稳定层次。

至少应维持以下边界：

- Host Layer：只负责宿主装配、命令入口、事件桥接、模块装载
- App Layer：负责用例编排，不承载底层技术细节
- Domain Layer：负责稳定产品语义、规则、契约、权限与能力模型
- Infra Layer：负责文件系统、sidecar、持久化、日志、网络等技术实现

如果这些层次继续混杂，就会带来两个问题：

- 功能边界难以判断，新增功能容易随意落点
- 所谓“并行开发低冲突”只是表面现象，实际会把复杂度重新压回公共层

### 2.2 公共模块与私有模块边界

必须明确哪些模块应该被大量依赖，哪些模块应该被限制在局部边界内。

这里需要特别说明：

- “避免大量依赖私有模块”不等于“模块之间不能协作”
- “保持私有边界”不等于“禁止功能联动”

真实系统里，功能联动本来就是常见场景。一个典型例子是：

- 后端新增了多个扩展能力
- AI 模块希望自然使用这些能力
- AI 因此需要看到知识图谱、搜索、文件操作、项目能力等其他扩展提供的功能

这个场景本身是合理的，问题不在于“AI 依赖了其他功能”，而在于“AI 是通过什么边界依赖它们”。

#### 应作为公共模块的内容

以下模块应当被设计成稳定公共依赖：

- 平台契约：command contract、event contract、capability contract、persistence contract
- 平台装配机制：模块注册、贡献收集、统一路由、权限校验入口
- 平台级共享能力：日志、trace、任务框架、基础错误模型、统一序列化约定
- 少量真正跨模块共享的 domain 抽象
- 面向跨模块协作的稳定 facade、registry 和 projection

这类公共模块的要求是：

- API 稳定
- 语义清晰
- 变更频率低
- 版本化意识强
- 不掺杂某个扩展的专有业务细节

#### 应作为私有模块的内容

以下内容应优先留在模块内部，避免被大量跨模块依赖：

- 扩展自己的 app service 实现细节
- 扩展自己的 infra adapter
- 扩展自己的持久化数据结构与清洗逻辑
- 扩展自己的 capability executor 具体实现
- 扩展自己的事件 payload 内部细节
- 扩展自己的测试 helper 和 fixture 组装逻辑

要点不是“这些模块永远不能被触达”，而是：

- 其他模块不应直接把这些实现细节当成长期公共 API
- 如果跨模块确实需要访问，应先上升为稳定 contract、capability 或 facade

这类私有模块的要求是：

- 尽量不被其他扩展直接依赖
- 如需对外暴露，优先暴露稳定 contract 或 facade，而不是内部实现
- 可以快速演进，但不应把变更传播到平台公共层

#### 协作依赖与实现依赖的区别

后端架构里应当允许“协作依赖”，但尽量避免“实现依赖”。

##### 合理的协作依赖

以下依赖通常是合理的：

- AI 模块依赖 capability registry，发现其他扩展暴露的能力
- AI 模块通过 capability/tool projection 调用其他扩展能力
- 一个扩展依赖平台定义的 shared contract 读取另一个扩展公开的结果
- 一个扩展依赖平台 facade 请求另一个模块执行稳定用例

这些依赖是面向“稳定协作边界”的依赖。

##### 不合理的实现依赖

以下依赖应尽量避免：

- AI 模块直接 import 另一个扩展的 app service 内部函数
- AI 模块直接依赖另一个扩展的 infra 实现
- 一个扩展直接读写另一个扩展的私有持久化结构
- 一个扩展基于另一个扩展的内部 DTO、内部事件 payload、内部 helper 编写功能

这些依赖会让两个模块在演进时互相绑死，并迅速制造并行开发冲突。

#### AI 复用其他扩展能力的推荐方式

以“AI 模块要使用其他扩展能力”为例，推荐边界如下：

1. 某个扩展把自己的能力注册为 capability 或稳定 facade
2. 平台统一维护 capability descriptor、权限、风险级别、输入输出契约
3. AI 模块只依赖 capability registry / tool projection / 平台 facade
4. AI 模块按 capability id 调用对应能力，而不是直接 import 对方实现
5. 能力执行由平台路由到对应扩展模块完成

这样一来，AI 可以广泛使用其他扩展能力，但不会因此直接依赖大量非基础模块的内部实现。

换句话说：

- AI 应依赖“平台公开的能力面”
- AI 不应依赖“其他扩展的私有实现面”

这也是后续判断“某项共享是否合理”的核心标准。

#### 判断规则

判断一个模块该是公共还是私有，可以用三个问题：

1. 它是否表达稳定平台语义，而不是某个扩展的临时实现？
2. 它是否真的需要被多个模块共享依赖？
3. 它被修改时，是否应当被视为平台级 breaking change？

如果答案大多是否定的，这个模块就不应被做成公共依赖。

如果一个模块确实需要被多个功能使用，还应再问一个问题：

4. 其他模块依赖它时，依赖的是“能力契约”，还是“内部实现细节”？

如果依赖的是内部实现细节，那么更合理的做法通常不是把它直接做成公共模块，而是提炼出新的公共 contract、capability 或 facade。

### 2.3 模块独立性

一个扩展功能的后端实现，理想情况下应主要落在自己的模块目录中完成，例如：

- 自己的 command 入口
- 自己的 app service
- 自己的 domain model
- 自己的 infra adapter
- 自己的测试
- 自己的私有持久化状态

新增功能不应默认要求同时修改：

- `lib.rs`
- 全局命令注册中心
- 全局 capability 大表
- 全局执行 `match` 分发器
- 全局事件名汇总文件
- 其他扩展的 app/domain/infra 模块

### 2.4 公共平台层稳定性

公共平台层应当是“低频变更层”，而不是“所有功能都要改的汇总层”。

理想状态下：

- 公共层定义稳定协议和扩展装配机制
- 扩展只需声明自己的贡献项
- 平台在启动时收集贡献项并完成注册

不理想状态下：

- 每新增一个功能，都要手改多个中央注册文件
- 每个能力都要加到一个超长 `match` 中
- 每个扩展都要共享一个持续膨胀的公共 DTO 或状态结构

### 2.5 冲突面最小化

如果三个 Agent 分别开发三个扩展功能，允许的冲突应主要发生在：

- 极少数稳定平台文件
- 明确的版本化契约文件
- 必要的依赖升级或基础设施演进点

不应普遍发生在：

- 命令注册
- capability 注册
- capability 执行
- 事件注册
- 全局状态结构
- 测试基座

### 2.6 扩展闭环能力

每个扩展应具备自己的后端闭环能力：

- 暴露自己的稳定接口
- 定义自己的状态与持久化
- 贡献自己的 capability 或 tool
- 拥有自己的后台任务与事件
- 拥有自己的集成测试样板

如果一个扩展的大部分改动都落在“公共层补洞”，说明当前架构仍未达到并行开发目标。

## 3. 基于新标准的当前结论

当前后端重构已经完成了重要基础，但还没有完全达到“合理分层、公共依赖稳定、私有模块边界清晰、并行开发低冲突”这组综合目标。

### 3.1 已经具备的基础

当前已经具备以下对并行开发有利的基础：

- 已形成 `host / app / domain / infra / runtime / shared` 分层
- Vault 的读写与查询能力已经收敛到 app/infra 层
- AI 设置、历史、sidecar、tool、callback 已经拆出独立模块
- capability registry 与 persistence contract 已经出现首版稳定模型
- extension private store 已经落地，为“扩展私有状态闭环”提供了基础设施

这些基础说明，后端已经不再是“所有功能往 `lib.rs` 和 commands 堆”的状态。

### 3.2 仍然阻碍并行开发的关键问题

虽然已经分层，但当前仍存在一些高冲突中心。

#### 1. 中央注册文件仍是高频修改点

当前新增命令、能力或入口时，仍然容易修改中央文件，例如：

- `host/command_registry.rs`
- `lib.rs` 中的对外导出聚合
- 全局 capability builtin 目录

这类文件会在多个功能并行推进时形成天然冲突点。

#### 2. Capability Registry 仍偏“中央清单 + 中央分发”

当前 capability 已经有注册中心，但内建能力仍主要集中在：

- 一个统一的 builtin 描述文件
- 一个统一的执行分发器

这意味着：

- 新增 capability 往往要改公共注册表
- 新增 capability 往往要改公共执行 `match`

这是典型的并行开发冲突源。

#### 3. 事件模型还不够模块化

当前事件通路存在，但还没有完全做到“每个扩展维护自己的事件契约并通过平台装配”。

如果未来多个扩展同时增加：

- 后端事件
- UI bridge 事件
- runtime 事件

那么仍可能争用同一批公共事件定义与桥接模块。

#### 4. 全局状态仍有继续膨胀的风险

`AppState` 目前还比较克制，但如果后续扩展都继续把运行时句柄塞进全局状态，就会重新形成共享热点。

这类问题的典型后果是：

- 新扩展都要改 `AppState`
- 初始化都要改 bootstrap
- 测试基座都要跟着改

#### 5. 后端扩展单元还没有完全成型

当前已经有“后端插件配置”“扩展私有存储”等基础，但还没有形成真正统一的后端扩展单元模型，例如：

- 扩展 manifest
- 扩展贡献项声明
- 扩展注册入口
- 扩展能力目录
- 扩展任务与事件边界

因此新增一个扩展时，仍更像“往平台里补一块功能”，而不是“接入一个边界明确的模块”。

#### 6. 跨模块协作边界还不够明确

当前已经开始建设 capability registry，这很好，但文档与实现层面还需要进一步统一一个原则：

- 允许模块之间通过稳定能力面协作
- 不鼓励模块之间直接依赖彼此私有实现

如果这个边界不明确，后续很容易出现一种表面上“功能联动自然”，实际上“AI 直接依赖很多扩展内部实现”的情况。

那样虽然短期开发快，但长期会带来：

- 模块间耦合迅速升高
- 公私边界失效
- 多 Agent 并行修改时冲突面扩大
- 任一扩展重构都容易波及 AI 模块

### 3.3 当前推进进度（2026-03-23）

已开始按本文档方向进行第一步后端改造。

当前已落地：

- Vault capability descriptors 已从中央 `domain/capability/builtin.rs` 提炼到 Vault 自己的 capability catalog 模块
- Vault capability 执行路由已从公共 `execution_app_service.rs` 提炼到 Vault 自己的 capability execution 模块
- 中央 capability catalog 与 execution 入口已开始改为“收集模块 contribution”而不是直接堆叠 Vault 细节
- 已建立统一的 backend module contribution 雏形，当前 Vault 与 AI 已能在同一模块贡献模型中声明命令、事件、持久化 owner 与 capability hooks
- 已将宿主平台命令也纳入统一 backend module contribution 视图，当前可统一描述 AI、Vault、Host Platform 三类模块的命令/事件/持久化 owner 元数据
- module contribution 已开始参与真实运行时校验：宿主持久化协议现在会校验 module_id 是否为已注册模块，以及 owner 是否在模块贡献中声明
- AI 模块事件元数据已改为复用宿主事件模块中的稳定事件常量，而不是在模块贡献中手写字符串
- 命令 ID 元数据已回收到各自 Tauri 命令模块中，module contribution 改为复用命令模块暴露的稳定清单，避免命令字符串在“宿主 handler 边界”和“模块元数据边界”双份漂移
- 事件 ID 元数据也开始回收到宿主事件模块中，AI module contribution 已复用事件模块暴露的稳定事件清单
- 已补充 backend module contribution 的基础一致性校验与单测，用于拦截空 ID、重复 command/event/owner 等低级漂移问题
- backend 启动前已接入一次 module contribution 自检，确保宿主装配在进入 Tauri builder 前就能暴露基础元数据冲突
- `host/command_registry.rs` 现在已为显式 `generate_handler!` 列表补充 command id 对照与一致性校验，宿主启动前会校验“显式 handler 命令表”和 module contribution 的命令声明没有漂移
- 宿主事件边界已补充与 module contribution 的一致性校验，启动前会校验“事件模块声明的稳定事件清单”和模块事件元数据没有偏差
- 模块事件元数据已从“只有事件 ID”升级为带分类的 event descriptor，当前平台已区分 `domain event / ui bridge event / runtime event`
- 宿主事件校验现在只接纳 `ui bridge` 事件进入前端桥接边界，非 UI 事件不再被错误纳入宿主显式事件注册范围
- 命令注册与事件边界都已补充针对缺失声明、游离注册、重复注册的单元测试，降低后续并行修改时的回归风险
- 已新增公开的 `test_support` 测试支持边界，并把集成测试与 bench 对 crate 根 `pub use` 的依赖迁移到该边界，`lib.rs` 不再继续承担大批量测试兼容出口聚合职责
- capability contribution 已补充启动前治理校验：现在不仅检查模块元数据，还会校验 capability descriptor 的基础完整性、catalog/execution 成对声明，以及“每个 descriptor 只能命中且必须命中一个模块执行路由”
- 已加入源码级模块边界守卫测试，明确禁止其他模块直接 import `app::vault`、`app::ai` 及其私有 infra 实现，避免后续再回到实现级耦合
- `VaultConfig` 已上移为 shared contract，消除了 host 命令层对 vault infra 持久化实现类型的直接依赖
- 已新增平台公共依赖面清单，并把 shared contracts、capability platform、persistence facade、host event bridge、module contribution、runtime state 等正式公共入口收敛为集中定义
- 架构守卫现在不仅检查“哪些私有实现不能被直接依赖”，也会检查“哪些公共依赖面只能被声明过的模块路径使用”，把公共边界从概念清单变成可执行规则
- 已新增模块边界模板层，AI/Vault 的私有命名空间规则已从架构守卫实现中抽成可复用模板；当前内建业务模块若未声明边界模板，会被测试直接拦截
- 已新增统一的 backend module manifest 入口，当前 AI、Vault、Host Platform 先通过 manifest 统一声明模块身份，module contribution 与边界模板开始从 manifest 派生，而不再完全分散维护

这一步的意义是：

- 新增 Vault 相关 capability 时，不再需要把描述细节和执行细节继续堆在同一个中央文件里
- capability 的契约与执行开始向“模块自持”演进，为后续做模块级 contribution registration 打基础
- 中央层开始具备“只收集 contribution，不了解模块实现细节”的雏形
- 后续再接入新模块时，已经有统一的 contribution 结构可扩展，而不需要为 capability、事件、持久化分别临时造新的中央约定
- 命令、事件、持久化 owner 现在已经有统一的模块贡献模型可挂载，后续可以继续把宿主自动装配逐步收敛到这层模型之上
- module contribution 不再只是描述性元数据，已经开始对持久化协议形成实际约束，这让“哪些 owner 属于哪个模块”从文档约定变成运行时规则
- 对 `tauri::generate_handler!` 的处理策略已明确调整为“保持原生显式 handler 列表，只把命令元数据下沉到模块本地维护”，避免为了进一步自动装配而偏离 Tauri 约定
- 对事件元数据也采用相同策略：事件发射边界继续留在宿主事件模块，module contribution 只复用事件模块声明出的稳定 ID 清单
- 第一优先级里与命令/事件边界相关的稳定性工作已完成：当前重点已经从“消除显式边界”转为“为显式边界补一致性守卫与回归测试”
- 事件平台层现在不再只是校验“有没有这个 ID”，而是开始校验“该事件是否属于正确的桥接类别”，这让未来继续扩展 domain/runtime 事件时不会误入前端桥接边界
- `lib.rs` 的职责已进一步收缩回“宿主入口 + 启动前校验 + test_support 模块暴露”，这让后续新增测试辅助能力不再默认修改后端入口文件
- capability 平台层已经从“只聚合”推进到“聚合 + 治理校验”，这让后续新增 capability 时不只是能接入，还必须满足统一平台规则
- 公共/私有边界已经不再只存在于文档描述中，而是开始通过 shared contract 上移和源码导入守卫形成可执行工程约束
- 平台公共依赖面现在也有了首版集中清单，后续新增跨模块协作入口时，需要先进入这份清单并通过守卫测试，避免公共层再次无序膨胀
- 模块私有边界现在也有了首版模板化入口，后续新增业务模块时不需要直接修改架构守卫实现，而是先补自己的边界模板与允许访问路径
- 模块接入平台现在也有了统一入口：后续新增业务模块时，应优先先补 manifest，再由平台派生 contribution、自检和边界模板，而不是分别改多处注册文件
- 已基于当前落地状态补充 v1《后端模块扩展标准流程》文档，后续新增模块应优先按 `docs/backend-module-extension-workflow.md` 执行，而不是继续依赖隐式口头约定

当前仍未完成：

- capability registry 仍然存在中央聚合入口，新增模块 contribution 仍需接入该聚合点
- capability 执行仍然由公共入口层统一调度，但已开始转为基于 contribution 的模块路由
- command/event/persistence owner 目前主要还是模块元数据，尚未完全驱动宿主自动装配
- `tauri::generate_handler!` 这条链路仍要求中央显式列出 handler 路径，当前不能像 capability 那样直接改为模块级 handler 宏拼接
- 命令注册层的目标已调整为“薄适配层 + 模块本地命令清单”，而不是继续尝试对 `generate_handler!` 做二次抽象
- module contribution 当前已经具备“最小一致性守卫”，后续可以继续按这个方向增加更贴近宿主装配的校验，而不是直接改造 Tauri 原生注册机制
- 平台公共依赖面当前是首版命名空间清单，还没有进一步收敛成更窄的 facade/projection 模块目录
- 事件分类治理已落地到模块 contribution 与宿主校验层，但每个模块的事件 payload 契约与命名空间规则还没有继续平台化
- 私有模块边界已具备首版模板化能力，但还没有继续和未来模块脚手架、目录生成器绑定起来
- 当前 manifest 已收敛 contribution 与边界模板，但公共依赖面规则仍然是平台级清单，还没有进一步并入单个模块 manifest 的声明模型
- 当前扩展流程文档已经可以指导长期开发，但仍属于 v1 实操规范，不是最终自动脚手架规范
- 这还只是“从中央大表抽出模块细节”的第一步，不是最终形态

## 4. 新目标：面向并行开发的后端目标状态

后端下一阶段的目标，不再只是继续拆文件，而是建立“分层稳定 + 公私边界明确 + 扩展贡献式”的架构。

### 4.1 目标状态

当一个新扩展接入后端时，理想工作流应接近：

1. 新建扩展自己的后端模块目录
2. 在模块内部声明自己的 command、event、capability、persistence、job
3. 通过一个稳定的贡献描述对象接入平台
4. 平台启动时自动收集并装配这些贡献
5. 绝大多数功能开发都不再修改中央公共文件

### 4.2 理想目录形态

后续应逐步从“按层集中大表”演进为“扩展内聚 + 平台装配”。

例如可以演进到类似结构：

```text
src-tauri/src/
  host/
    bootstrap/
    registry/
    platform/

  modules/
    ai_chat/
      host/
      app/
      domain/
      infra/
      tests/
      module.rs
    knowledge_graph/
      host/
      app/
      domain/
      infra/
      tests/
      module.rs
    custom_activity/
      host/
      app/
      domain/
      infra/
      tests/
      module.rs

  runtime/
  shared/
  platform/
```

这里的关键不是目录名字，而是原则：

- 平台层定义装配协议
- 模块层负责自己的能力闭环

## 5. 接下来还要做的工作

下面的工作项以“先稳住分层和公共依赖边界，再降低并行开发冲突”为排序依据，而不是按技术炫技或局部整洁度排序。

### 5.1 第一优先级：先稳住公共模块边界

这部分工作先回答“什么应该被依赖、什么不应该被依赖”，否则后续即使拆掉中央热点，也只是把冲突转移到别处。

#### 工作 1：明确平台公共模块清单

目标：正式定义哪些模块属于平台公共依赖，后续新增功能应优先依赖这些稳定边界，而不是绕过它们直接依赖内部实现。

至少应明确：

- 哪些 contract 属于 shared/platform 层
- 哪些 registry/route/permission 属于平台层
- 哪些 runtime 能力属于所有扩展共享
- 哪些 facade 是允许跨模块依赖的正式入口
- 哪些 capability/projection 是 AI 等协作模块的正式依赖面

达成后收益：

- 后续扩展知道“应该依赖什么”
- 公共层的职责收敛，不再无限长大

#### 工作 2：明确模块私有边界与禁止依赖规则

目标：正式定义哪些目录和模块默认不应该被其他扩展大量依赖。

至少应约束：

- 扩展内部 app service 不作为公共 API
- 扩展内部 infra 实现不作为公共依赖
- 扩展内部持久化结构不直接被其他模块复用
- 扩展内部测试 helper 不向外扩散
- AI 等协作模块默认不得直接依赖其他扩展的私有 app/infra 实现

达成后收益：

- 减少“先图方便，后面变成公共历史包袱”的情况
- 私有模块可以更自由演进

#### 工作 3：定义跨模块协作的标准通路

目标：明确后端功能联动时，优先通过哪些稳定边界协作，而不是让调用方自己决定直接依赖哪个实现。

至少应明确以下通路：

- capability registry
- tool projection
- platform facade
- stable shared contracts
- event-based coordination

达成后收益：

- AI 使用其他扩展能力会成为架构内的正常路径
- 功能联动增加时，不会自然滑向实现级耦合

### 5.2 第二优先级：消除中央高冲突点

这部分工作最直接决定三个 Agent 能否同时做三个扩展而少冲突。

#### 工作 4：消除中央 capability 描述大表

目标：每个模块在自己目录下声明 capability，而不是把所有 capability 写进同一个 builtin 文件。

需要完成：

- 为模块定义 capability contribution 接口
- 每个模块返回自己的 capability descriptors
- 平台在启动时聚合所有 descriptors
- 避免新增功能时继续修改单一 builtin 列表

达成后收益：

- 两个扩展同时新增 capability，不必改同一个文件
- capability 描述与功能实现保持就近维护

#### 工作 5：消除中央 capability 执行分发器

目标：新增 capability 时，不再编辑一个持续膨胀的全局 `match`。

需要完成：

- 定义 capability executor 注册接口
- executor 随模块一起注册
- 平台按 capability id 路由到对应模块 executor

达成后收益：

- 多个 Agent 同时加 capability 时不会争用同一执行器文件
- 注册和实现可以保持在模块内部闭环

#### 工作 6：降低中央命令注册冲突

目标：新增后端命令时，不再总是手动修改统一命令注册表。

可接受的过渡方案：

- 继续保留集中装配，但引入模块级命令列表
- 中央层只聚合模块贡献，而不直接知道每个具体命令细节

更理想的方案：

- 构建后端模块 manifest 或 contribution object
- 命令由模块声明，平台统一收集

#### 工作 7：收缩 `lib.rs` 的兼容性聚合职责

目标：避免后续新增功能继续在 `lib.rs` 上叠加 `pub use` 出口。

需要完成：

- 区分“运行时宿主入口”和“测试支持出口”
- 为测试建立更稳定的测试辅助模块或 facade
- 减少扩展开发对 `lib.rs` 的直接编辑需求

### 5.3 第三优先级：形成真正的后端扩展单元

#### 工作 8：定义后端模块/扩展贡献模型

目标：让新增扩展不再是“修改平台代码”，而是“向平台注册模块”。

贡献模型至少应覆盖：

- `module_id`
- `api_version`
- commands
- events
- capabilities
- persistence owners
- background jobs
- 依赖的 platform permissions

达成后，平台的职责变成：

- 装配模块
- 校验冲突
- 提供统一治理

而不是手工维护所有扩展的细节。

#### 工作 9：为每个扩展建立独立的后端边界模板

目标：让新增扩展时有固定脚手架，不需要自行决定该改哪些公共层。

模板应至少包含：

- 模块级文档注释
- command 入口
- app service
- domain model
- infra adapter
- capability contribution
- persistence owner
- 测试文件

这样可以把“架构决策”前置到模板，而不是让每次新增功能都重新讨论边界。

### 5.4 第四优先级：继续拆解共享热点

#### 工作 10：限制 `AppState` 的扩张方式

目标：避免所有扩展共享一个持续膨胀的状态结构。

建议原则：

- `AppState` 仅保留平台级共享句柄
- 模块级运行时句柄优先放在模块自己的 runtime/state 容器中
- 如必须放入全局状态，需要满足明确的跨模块共享理由

#### 工作 11：建立模块级事件契约边界

目标：新增事件主要改自己的模块，而不是改一个公共事件总表。

需要完成：

- 约束事件命名空间
- 区分 domain event / ui bridge event / runtime event
- 为每个模块维护自己的事件 payload 契约
- 平台层只负责发射机制和公共约束

#### 工作 12：把共享 DTO 控制在稳定协议层

目标：共享结构体只保留真正的平台协议，不承载扩展内部细节。

否则后续会出现：

- 一个 shared contracts 文件被多个扩展反复修改
- 一个字段变化影响多条功能线并制造冲突

### 5.5 第五优先级：建立可并行的任务与测试基础设施

#### 工作 13：建立统一 job/task 平台，但按模块归属执行

目标：后台任务要统一治理，但定义和实现尽量在模块内部。

统一平台负责：

- task id
- trace id
- progress
- cancellation
- retry
- concurrency control

模块负责：

- 声明任务类型
- 实现任务逻辑
- 提供任务结果语义

这样既能统一治理，又不会让所有任务代码挤在同一公共模块里。

#### 工作 14：建立模块级集成测试样板

目标：每个扩展都能独立补集成测试，而不是共享一套容易冲突的大型测试文件。

应推动：

- 每个模块有自己的 integration test 入口
- 每个模块有自己的 fixtures 或 fixture 组合方式
- 通用测试 helper 稳定后尽量少改

### 5.6 第六优先级：建立并行开发的工程约束

#### 工作 15：定义“公共层变更门槛”

需要在工程上明确：

- 什么情况下允许修改平台公共层
- 什么情况下必须优先在模块内闭环解决
- 哪些文件被视为高冲突文件，需要额外评审

否则即使架构上可拆，开发习惯仍会让改动重新回流到中央层。

#### 工作 16：为公共注册点建立冲突预警规则

例如可约束：

- 新 capability 默认不得直接写入中央 builtin 表
- 新事件默认不得先加进全局事件总表
- 新持久化状态默认不得直接塞入共享 config

这类规则的价值在于保护已经建立的模块边界。

## 6. 推荐实施顺序

基于“合理分层 + 公私边界清晰 + 并行开发低冲突”目标，推荐按以下顺序推进。

### 阶段 A：先定义公共层与私有层边界

先做：

- 平台公共模块清单
- 私有模块禁止依赖规则
- 跨模块协作标准通路
- 共享 facade 与 contract 范围定义
- `AppState` 与 shared contracts 的边界约束

这一阶段的目标是先回答“以后什么可以共享，什么不能共享”。

### 阶段 B：再拆冲突热点

先做：

- capability descriptor 的模块化贡献
- capability executor 的模块化注册
- 命令注册的模块化聚合
- `lib.rs` 测试导出收缩

这一阶段的目标不是做出完整扩展平台，而是把最容易冲突的中央文件降温。

### 阶段 C：再建立后端模块单元

再做：

- 后端模块 contribution model
- 模块脚手架模板
- 模块级事件契约
- 模块级 persistence owner 声明

这一阶段完成后，新增扩展应能主要在自身目录内落地。

### 阶段 D：最后补统一治理层

最后做：

- 统一 task/job runtime
- 权限与确认平台化
- 审计与 trace 平台化
- 扩展存储迁移框架

这部分仍重要，但它们对“减少并行修改冲突”的直接收益小于前两阶段，因此不应排在最前。

## 7. 达标判断

当后端满足以下条件时，可以认为已经基本达到“多 Agent 并行开发、低冲突”的目标：

1. 平台公共模块清单稳定，并且语义边界清晰
2. 扩展私有模块默认不被其他扩展直接大量依赖
3. AI 等协作模块依赖的是平台公开的 capability/facade，而不是其他扩展私有实现
4. 三个扩展同时新增后端能力时，主要改动都在各自模块目录中
5. 新增 capability 不需要修改中央 builtin 表和中央执行 `match`
6. 新增命令不需要频繁修改统一命令注册文件
7. 新增事件和持久化状态主要在模块内声明，不侵入公共总表
8. `AppState` 不再成为所有扩展共享追加字段的位置
9. 测试新增主要落在模块自己的测试文件中
10. 平台公共层改动频率显著低于扩展层改动频率

如果这些条件还做不到，就说明当前架构仍然更适合“单线重构”，而不是“多 Agent 并行开发”。

## 8. 一句话结论

后端下一阶段的重点，不是阻止功能联动，而是让功能联动建立在稳定协作边界之上：公共层提供 contract、capability 和 facade，私有层保留实现细节，新增功能尽量只修改自己的模块，而不是通过直接依赖别的扩展内部实现来扩散耦合。
