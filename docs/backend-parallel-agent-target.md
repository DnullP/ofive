# ofive 后端并行开发架构目标与后续工作

## 1. 文档目的

本文档用于替代旧版“后端目标架构设计”作为当前后端规划依据。

当前后端规划同时关注几类目标：

- 合理分层
- 明确模块职责
- 区分公共模块与私有模块
- 支持多个 Agent 并行开发时尽量降低冲突
- 让公共依赖保持稳定、可治理、可版本化

其中，“多个 Agent 并行开发时不轻易发生代码冲突”只是评估标准之一，不是唯一标准。
新的规划要求后端最终能够同时满足：

- 多个 Agent 并行开发不同扩展功能
- 不轻易发生对同一公共模块的同时修改
- 新增功能尽量在各自边界内闭环完成
- 公共平台层保持稳定、低频变更、可版本化

换句话说，后端架构是否足够好，不能只看“模块是否拆开”，而要同时看：

- 分层是否合理
- 哪些模块应该作为稳定公共依赖
- 哪些模块应该作为扩展私有实现而避免被大量依赖
- 三个扩展功能能否同时开发
- 每个功能是否主要修改自己的模块边界
- 公共文件是否只在建立平台能力时少量变更，而不是每次扩展都要反复编辑

## 2. 核心评估标准

以后端是否同时满足“合理分层 + 清晰依赖边界 + 可并行开发”为准，采用以下标准判断架构是否达标。

### 2.1 分层合理性

后端分层仍然是第一原则之一，因为并行开发能力本身依赖于稳定层次。

至少应维持以下边界：

- Host Layer：只负责宿主装配、命令入口、事件桥接、模块装载
- App Layer：负责用例编排，不承载底层技术细节
- Domain Layer：负责稳定产品语义、规则、契约、权限与能力模型
- Infra Layer：负责文件系统、sidecar、持久化、日志、网络等技术实现

如果这些层次继续混杂，就会带来两个问题：

- 功能边界难以判断，新增功能容易随意落点
- 所谓“并行开发低冲突”只是表面现象，实际会把复杂度重新压回公共层

### 2.2 公共模块与私有模块边界

必须明确哪些模块应该被大量依赖，哪些模块应该被限制在局部边界内。

这里需要特别说明：

- “避免大量依赖私有模块”不等于“模块之间不能协作”
- “保持私有边界”不等于“禁止功能联动”

真实系统里，功能联动本来就是常见场景。一个典型例子是：

- 后端新增了多个扩展能力
- AI 模块希望自然使用这些能力
- AI 因此需要看到知识图谱、搜索、文件操作、项目能力等其他扩展提供的功能

这个场景本身是合理的，问题不在于“AI 依赖了其他功能”，而在于“AI 是通过什么边界依赖它们”。

#### 应作为公共模块的内容

以下模块应当被设计成稳定公共依赖：

- 平台契约：command contract、event contract、capability contract、persistence contract
- 平台装配机制：模块注册、贡献收集、统一路由、权限校验入口
- 平台级共享能力：日志、trace、任务框架、基础错误模型、统一序列化约定
- 少量真正跨模块共享的 domain 抽象
- 面向跨模块协作的稳定 facade、registry 和 projection

这类公共模块的要求是：

- API 稳定
- 语义清晰
- 变更频率低
- 版本化意识强
- 不掺杂某个扩展的专有业务细节

#### 应作为私有模块的内容

以下内容应优先留在模块内部，避免被大量跨模块依赖：

- 扩展自己的 app service 实现细节
- 扩展自己的 infra adapter
- 扩展自己的持久化数据结构与清洗逻辑
- 扩展自己的 capability executor 具体实现
- 扩展自己的事件 payload 内部细节
- 扩展自己的测试 helper 和 fixture 组装逻辑

要点不是“这些模块永远不能被触达”，而是：

- 其他模块不应直接把这些实现细节当成长期公共 API
- 如果跨模块确实需要访问，应先上升为稳定 contract、capability 或 facade

这类私有模块的要求是：

- 尽量不被其他扩展直接依赖
- 如需对外暴露，优先暴露稳定 contract 或 facade，而不是内部实现
- 可以快速演进，但不应把变更传播到平台公共层

#### 协作依赖与实现依赖的区别

后端架构里应当允许“协作依赖”，但尽量避免“实现依赖”。

##### 合理的协作依赖

以下依赖通常是合理的：

- AI 模块依赖 capability registry，发现其他扩展暴露的能力
- AI 模块通过 capability/tool projection 调用其他扩展能力
- 一个扩展依赖平台定义的 shared contract 读取另一个扩展公开的结果
- 一个扩展依赖平台 facade 请求另一个模块执行稳定用例

这些依赖是面向“稳定协作边界”的依赖。

##### 不合理的实现依赖

以下依赖应尽量避免：

- AI 模块直接 import 另一个扩展的 app service 内部函数
- AI 模块直接依赖另一个扩展的 infra 实现
- 一个扩展直接读写另一个扩展的私有持久化结构
- 一个扩展基于另一个扩展的内部 DTO、内部事件 payload、内部 helper 编写功能

这些依赖会让两个模块在演进时互相绑死，并迅速制造并行开发冲突。

#### AI 复用其他扩展能力的推荐方式

以“AI 模块要使用其他扩展能力”为例，推荐边界如下：

1. 某个扩展把自己的能力注册为 capability 或稳定 facade
2. 平台统一维护 capability descriptor、权限、风险级别、输入输出契约
3. AI 模块只依赖 capability registry / tool projection / 平台 facade
4. AI 模块按 capability id 调用对应能力，而不是直接 import 对方实现
5. 能力执行由平台路由到对应扩展模块完成

这样一来，AI 可以广泛使用其他扩展能力，但不会因此直接依赖大量非基础模块的内部实现。

换句话说：

- AI 应依赖“平台公开的能力面”
- AI 不应依赖“其他扩展的私有实现面”

这也是后续判断“某项共享是否合理”的核心标准。

#### 判断规则

判断一个模块该是公共还是私有，可以用三个问题：

1. 它是否表达稳定平台语义，而不是某个扩展的临时实现？
2. 它是否真的需要被多个模块共享依赖？
3. 它被修改时，是否应当被视为平台级 breaking change？

如果答案大多是否定的，这个模块就不应被做成公共依赖。

如果一个模块确实需要被多个功能使用，还应再问一个问题：

4. 其他模块依赖它时，依赖的是“能力契约”，还是“内部实现细节”？

如果依赖的是内部实现细节，那么更合理的做法通常不是把它直接做成公共模块，而是提炼出新的公共 contract、capability 或 facade。

### 2.3 模块独立性

一个扩展功能的后端实现，理想情况下应主要落在自己的模块目录中完成，例如：

- 自己的 command 入口
- 自己的 app service
- 自己的 domain model
- 自己的 infra adapter
- 自己的测试
- 自己的私有持久化状态

新增功能不应默认要求同时修改：

- `lib.rs`
- 全局命令注册中心
- 全局 capability 大表
- 全局执行 `match` 分发器
- 全局事件名汇总文件
- 其他扩展的 app/domain/infra 模块

### 2.4 公共平台层稳定性

公共平台层应当是“低频变更层”，而不是“所有功能都要改的汇总层”。

理想状态下：

- 公共层定义稳定协议和扩展装配机制
- 扩展只需声明自己的贡献项
- 平台在启动时收集贡献项并完成注册

不理想状态下：

- 每新增一个功能，都要手改多个中央注册文件
- 每个能力都要加到一个超长 `match` 中
- 每个扩展都要共享一个持续膨胀的公共 DTO 或状态结构

### 2.5 冲突面最小化

如果三个 Agent 分别开发三个扩展功能，允许的冲突应主要发生在：

- 极少数稳定平台文件
- 明确的版本化契约文件
- 必要的依赖升级或基础设施演进点

不应普遍发生在：

- 命令注册
- capability 注册
- capability 执行
- 事件注册
- 全局状态结构
- 测试基座

### 2.6 扩展闭环能力

每个扩展应具备自己的后端闭环能力：

- 暴露自己的稳定接口
- 定义自己的状态与持久化
- 贡献自己的 capability 或 tool
- 拥有自己的后台任务与事件
- 拥有自己的集成测试样板

如果一个扩展的大部分改动都落在“公共层补洞”，说明当前架构仍未达到并行开发目标。

## 3. 基于新标准的当前结论

当前后端重构已经完成了重要基础，但还没有完全达到“合理分层、公共依赖稳定、私有模块边界清晰、并行开发低冲突”这组综合目标。

### 3.1 已经具备的基础

当前已经具备以下对并行开发有利的基础：

- 已形成 `host / app / domain / infra / runtime / shared` 分层
- Vault 的读写与查询能力已经收敛到 app/infra 层
- AI 设置、历史、sidecar、tool、callback 已经拆出独立模块
- capability registry 与 persistence contract 已经出现首版稳定模型
- extension private store 已经落地，为“扩展私有状态闭环”提供了基础设施

这些基础说明，后端已经不再是“所有功能往 `lib.rs` 和 commands 堆”的状态。

### 3.2 仍然阻碍并行开发的关键问题

虽然已经分层，但当前仍存在一些高冲突中心。

#### 1. 中央注册文件仍是高频修改点

当前新增命令、能力或入口时，仍然容易修改中央文件，例如：

- `host/command_registry.rs`
- `lib.rs` 中的对外导出聚合
- 全局 capability builtin 目录

这类文件会在多个功能并行推进时形成天然冲突点。

#### 2. Capability Registry 仍偏“中央清单 + 中央分发”

当前 capability 已经有注册中心，但内建能力仍主要集中在：

- 一个统一的 builtin 描述文件
- 一个统一的执行分发器

这意味着：

- 新增 capability 往往要改公共注册表
- 新增 capability 往往要改公共执行 `match`

这是典型的并行开发冲突源。

#### 3. 事件模型还不够模块化

当前事件通路存在，但还没有完全做到“每个扩展维护自己的事件契约并通过平台装配”。

如果未来多个扩展同时增加：

- 后端事件
- UI bridge 事件
- runtime 事件

那么仍可能争用同一批公共事件定义与桥接模块。

#### 4. 全局状态仍有继续膨胀的风险

`AppState` 目前还比较克制，但如果后续扩展都继续把运行时句柄塞进全局状态，就会重新形成共享热点。

这类问题的典型后果是：

- 新扩展都要改 `AppState`
- 初始化都要改 bootstrap
- 测试基座都要跟着改

#### 5. 后端扩展单元还没有完全成型

当前已经有“后端插件配置”“扩展私有存储”等基础，但还没有形成真正统一的后端扩展单元模型，例如：

- 扩展 manifest
- 扩展贡献项声明
- 扩展注册入口
- 扩展能力目录
- 扩展任务与事件边界

因此新增一个扩展时，仍更像“往平台里补一块功能”，而不是“接入一个边界明确的模块”。

#### 6. 跨模块协作边界还不够明确

当前已经开始建设 capability registry，这很好，但文档与实现层面还需要进一步统一一个原则：

- 允许模块之间通过稳定能力面协作
- 不鼓励模块之间直接依赖彼此私有实现

如果这个边界不明确，后续很容易出现一种表面上“功能联动自然”，实际上“AI 直接依赖很多扩展内部实现”的情况。

那样虽然短期开发快，但长期会带来：

- 模块间耦合迅速升高
- 公私边界失效
- 多 Agent 并行修改时冲突面扩大
- 任一扩展重构都容易波及 AI 模块

### 3.3 当前推进进度（2026-03-23）

已开始按本文档方向进行第一步后端改造。

当前已落地：

- Vault capability descriptors 已从中央 `domain/capability/builtin.rs` 提炼到 Vault 自己的 capability catalog 模块
- Vault capability 执行路由已从公共 `execution_app_service.rs` 提炼到 Vault 自己的 capability execution 模块
- 中央 capability catalog 与 execution 入口已开始改为“收集模块 contribution”而不是直接堆叠 Vault 细节
- 已建立统一的 backend module contribution 雏形，当前 Vault 与 AI 已能在同一模块贡献模型中声明命令、事件、持久化 owner 与 capability hooks
- 已将宿主平台命令也纳入统一 backend module contribution 视图，当前可统一描述 AI、Vault、Host Platform 三类模块的命令/事件/持久化 owner 元数据
- module contribution 已开始参与真实运行时校验：宿主持久化协议现在会校验 module_id 是否为已注册模块，以及 owner 是否在模块贡献中声明
- AI 模块事件元数据已改为复用宿主事件模块中的稳定事件常量，而不是在模块贡献中手写字符串
- 命令 ID 元数据已回收到各自 Tauri 命令模块中，module contribution 改为复用命令模块暴露的稳定清单，避免命令字符串在“宿主 handler 边界”和“模块元数据边界”双份漂移
- 事件 ID 元数据也开始回收到宿主事件模块中，AI module contribution 已复用事件模块暴露的稳定事件清单
- 已补充 backend module contribution 的基础一致性校验与单测，用于拦截空 ID、重复 command/event/owner 等低级漂移问题
- backend 启动前已接入一次 module contribution 自检，确保宿主装配在进入 Tauri builder 前就能暴露基础元数据冲突
- `host/command_registry.rs` 现在已为显式 `generate_handler!` 列表补充 command id 对照与一致性校验，宿主启动前会校验“显式 handler 命令表”和 module contribution 的命令声明没有漂移
- 宿主事件边界已补充与 module contribution 的一致性校验，启动前会校验“事件模块声明的稳定事件清单”和模块事件元数据没有偏差
- 模块事件元数据已从“只有事件 ID”升级为带分类的 event descriptor，当前平台已区分 `domain event / ui bridge event / runtime event`
- 宿主事件校验现在只接纳 `ui bridge` 事件进入前端桥接边界，非 UI 事件不再被错误纳入宿主显式事件注册范围
- 命令注册与事件边界都已补充针对缺失声明、游离注册、重复注册的单元测试，降低后续并行修改时的回归风险
- 已新增公开的 `test_support` 测试支持边界，并把集成测试与 bench 对 crate 根 `pub use` 的依赖迁移到该边界，`lib.rs` 不再继续承担大批量测试兼容出口聚合职责
- capability contribution 已补充启动前治理校验：现在不仅检查模块元数据，还会校验 capability descriptor 的基础完整性、catalog/execution 成对声明，以及“每个 descriptor 只能命中且必须命中一个模块执行路由”
- 已加入源码级模块边界守卫测试，明确禁止其他模块直接 import `app::vault`、`app::ai` 及其私有 infra 实现，避免后续再回到实现级耦合
- `VaultConfig` 已上移为 shared contract，消除了 host 命令层对 vault infra 持久化实现类型的直接依赖
- 已新增平台公共依赖面清单，并把 shared contracts、capability platform、persistence facade、host event bridge、module contribution、runtime state 等正式公共入口收敛为集中定义
- 架构守卫现在不仅检查“哪些私有实现不能被直接依赖”，也会检查“哪些公共依赖面只能被声明过的模块路径使用”，把公共边界从概念清单变成可执行规则
- 已新增模块边界模板层，AI/Vault 的私有命名空间规则已从架构守卫实现中抽成可复用模板；当前内建业务模块若未声明边界模板，会被测试直接拦截
- 已新增统一的 backend module manifest 入口，当前 AI、Vault、Host Platform 先通过 manifest 统一声明模块身份，module contribution 与边界模板开始从 manifest 派生，而不再完全分散维护

这一步的意义是：

- 新增 Vault 相关 capability 时，不再需要把描述细节和执行细节继续堆在同一个中央文件里
- capability 的契约与执行开始向“模块自持”演进，为后续做模块级 contribution registration 打基础
- 中央层开始具备“只收集 contribution，不了解模块实现细节”的雏形
- 后续再接入新模块时，已经有统一的 contribution 结构可扩展，而不需要为 capability、事件、持久化分别临时造新的中央约定
- 命令、事件、持久化 owner 现在已经有统一的模块贡献模型可挂载，后续可以继续把宿主自动装配逐步收敛到这层模型之上
- module contribution 不再只是描述性元数据，已经开始对持久化协议形成实际约束，这让“哪些 owner 属于哪个模块”从文档约定变成运行时规则
- 对 `tauri::generate_handler!` 的处理策略已明确调整为“保持原生显式 handler 列表，只把命令元数据下沉到模块本地维护”，避免为了进一步自动装配而偏离 Tauri 约定
- 对事件元数据也采用相同策略：事件发射边界继续留在宿主事件模块，module contribution 只复用事件模块声明出的稳定 ID 清单
- 第一优先级里与命令/事件边界相关的稳定性工作已完成：当前重点已经从“消除显式边界”转为“为显式边界补一致性守卫与回归测试”
- 事件平台层现在不再只是校验“有没有这个 ID”，而是开始校验“该事件是否属于正确的桥接类别”，这让未来继续扩展 domain/runtime 事件时不会误入前端桥接边界
- `lib.rs` 的职责已进一步收缩回“宿主入口 + 启动前校验 + test_support 模块暴露”，这让后续新增测试辅助能力不再默认修改后端入口文件
- capability 平台层已经从“只聚合”推进到“聚合 + 治理校验”，这让后续新增 capability 时不只是能接入，还必须满足统一平台规则
- 公共/私有边界已经不再只存在于文档描述中，而是开始通过 shared contract 上移和源码导入守卫形成可执行工程约束
- 平台公共依赖面现在也有了首版集中清单，后续新增跨模块协作入口时，需要先进入这份清单并通过守卫测试，避免公共层再次无序膨胀
- 模块私有边界现在也有了首版模板化入口，后续新增业务模块时不需要直接修改架构守卫实现，而是先补自己的边界模板与允许访问路径
- 模块接入平台现在也有了统一入口：后续新增业务模块时，应优先先补 manifest，再由平台派生 contribution、自检和边界模板，而不是分别改多处注册文件
- 已基于当前落地状态补充 v1《后端模块扩展标准流程》文档，后续新增模块应优先按 `docs/backend-module-extension-workflow.md` 执行，而不是继续依赖隐式口头约定

当前保留的架构边界：

- capability registry 仍保留统一聚合入口，但聚合内容已经完全从模块 contribution 派生；该入口现在属于平台治理面，而不是业务扩展冲突点
- capability 执行仍由公共入口层统一调度，但实际执行路由已收敛为模块 contribution 驱动；该公共入口承担的是平台路由职责
- `tauri::generate_handler!` 仍要求显式列出 handler 路径；当前将其保留为宿主原生显式装配边界，而不是继续抽象成隐式宏拼接
- 命令、事件、持久化 owner 的装配仍采用“模块元数据 + 启动前一致性校验”的模式；当前把它定义为稳定平台约束，而不是继续追求更强自动装配
- 平台公共依赖面现已从内建模块 manifest 派生，公共规则不再单独维护中央清单；平台继续保留公共面守卫测试作为治理手段
- 模块扩展流程当前以文档化模板和工程守卫为准，不再把“脚手架生成器”作为本次重构的交付目标
